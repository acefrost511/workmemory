# Claude Code 架构深度解析

## 一、整体架构概览

Claude Code 是一个高度成熟的 AI 编程助手，其架构围绕以下核心维度：

```
Claude Code 架构
├── 1. 多层提示词系统（57个文件，324条提示词）
├── 2. 记忆体系（短期 + 长期 + 做梦整合）
├── 3. 多Agent协作（Fork / SubAgent / Team / Swarm）
├── 4. 上下文管理（自动压缩 / 会话记忆 / 无限上下文）
├── 5. 工具系统（40+ 工具，权限分级）
└── 6. 文件体系（CLAUDE.md / 记忆目录 / MagicDocs）
```

## 二、记忆体系（四层架构）

### 2.1 工作记忆（Working Memory）
- 载体：当前对话上下文
- 特性：自动压缩，截断最早的非关键内容

### 2.2 会话记忆（SessionMemory）
- 10个固定板块，跨会话恢复当前工作状态
- 每次新会话开始时自动读取

### 2.3 持久记忆（memory/目录）
- 永久存储，分为主题记忆和日志
- MEMORY.md 作为索引（≤200行）

### 2.4 做梦机制（Dream）
- 定时触发，四阶段：建立方向感→收集信号→整合→修剪索引
- 本质是记忆维护，不是创意工厂

## 三、做梦机制详解

### 目的
解决 AI 的"遗忘问题"：记忆文件膨胀、MEMORY.md 截断、新会话方向感丢失。

### 四阶段

| 阶段 | 做什么 | 本质 |
|------|--------|------|
| 1. 建立方向感 | ls目录、读MEMORY.md、浏览已有文件 | "我现在记得什么？" |
| 2. 收集近期信号 | 读日志、检查矛盾、grep转录 | "最近发生了什么？" |
| 3. 整合 | 合并到已有文件、修正矛盾、删除过时 | "更新记忆，不制造重复" |
| 4. 修剪索引 | 控制MEMORY.md≤200行、缩短过长条目 | "保持索引精简可用" |

### 关键设计哲学
- 不做穷尽式通读，只 grep 重要的内容
- 更新优于新建，合并到已有主题文件
- 主动修正错误，在源头修正不追加
- MEMORY.md 只放索引 hook，不放内容

## 四、记忆管理体系

### 三层架构

```
工作记忆（对话上下文）  ← 单次会话，自动压缩
会话记忆（10个固定板块）← 跨会话恢复当前工作
持久记忆（memory/目录） ← 永久，Dream维护
```

### 两步保存流程
1. 写记忆文件（内容）
2. 更新 MEMORY.md 索引

### 记忆文件格式（frontmatter）

```yaml
---
name: 记忆名称
description: 一句话描述
type: fact/preference/convention
---
记忆正文...
```

## 五、文件管理体系

### CLAUDE.md 多层级
- 项目根目录/CLAUDE.md ← 全项目通用指导
- 子目录/CLAUDE.md ← 子目录特定指导
- ~/.claude/CLAUDE.md ← 用户全局偏好

### MagicDocs
对话中有新的学习/洞察时，自动维护的文档文件。

## 六、多Agent架构

### Agent 类型
- **fork**：后台并行，上下文隔离（"射后不管"）
- **subagent**：专业子Agent（explore/verify）
- **teammate**：团队协作，通过 SendMessage 通信

### 独立验证
非琐碎实现（3+文件修改、后端变更）必须经过独立对抗性验证。

## 七、对总管的落地评估

### ✅ 可直接实现
- 每日对话日志（logs/YYYY/YYYY-MM-DD.md）
- MEMORY.md 索引化（≤200行）
- 两步保存流程
- 快速检索（grep日志）

### ⚠️ 部分实现
- 做梦机制（Automation定时触发）
- 记忆抽取子Agent

### ❌ 无法实现
- 独立后台服务
- JSONL转录存档
- fork上下文隔离分支

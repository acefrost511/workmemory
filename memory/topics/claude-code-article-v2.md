# Claude Code 架构深度解析：记忆系统、做梦机制与多 Agent 协作

> 逆向 Anthropic 旗下 AI 编程助手 Claude Code 的 324 条系统提示词，从中提炼出值得所有 AI 应用借鉴的架构设计。
> 本文面向所有 AI 应用开发者和使用者，部分段落附有通俗解读，帮助非专业读者理解设计背后的原理。

---

## 前言

Claude Code 是 Anthropic 推出的 AI 编程助手，与 Cursor、GitHub Copilot 等产品定位类似。它的独特之处在于：系统提示词被完整逆向并翻译为中文，共计 324 条、112 页——这为理解一个顶级 AI 产品的内部设计提供了极其罕见的窗口。

本文从这 324 条提示词中提炼出三大核心架构——**记忆系统**、**做梦机制**、**多 Agent 协作**，以第三方的视角进行分析和总结，期望为从事 AI 应用开发的读者提供参考。

---

## 一、整体架构概览

Claude Code 的系统由主 Agent、记忆系统、工具箱和子 Agent 四大部分组成：

```
┌─────────────────────────────────────────────────────────────┐
│                        主 Agent                              │
│  （直接与用户对话的核心智能体）                                │
│                                                              │
│  ├── 记忆系统（知道你是谁、做过什么）                          │
│  ├── 工具箱（读文件、写代码、搜索、运行命令...）                │
│  └── 子 Agent（派出去干活的助手）                              │
│                                                              │
│        │                                                     │
│        ├── fork（后台分身，上下文隔离，用完即销毁）             │
│        ├── subagent（专业助手：代码探索、代码审查）             │
│        └── teammate（团队成员：多人协作）                      │
└─────────────────────────────────────────────────────────────┘
```

### 🎯 通俗理解

你可以把 Claude Code 想象成一个**AI 公司**：

- **主 Agent** = 总经理，直接和客户（用户）对话，了解需求、分配任务
- **记忆系统** = 公司档案室，存放所有客户资料、项目历史、工作规范
- **工具箱** = 员工的工具包，螺丝刀、锤子、电脑等
- **子 Agent** = 外派的专业顾问，需要时派出去专项解决问题

与记忆系统和多 Agent 协作相比，工具箱部分（文件读写、命令执行等）相对常规。以下重点剖析前两者。

## 二、记忆系统——三层架构与图书馆式组织

### 2.1 三层记忆

Claude Code 将记忆划分为三个层级，各层职责清晰：

| 层级 | 名称 | 生命周期 | 用途 | 类比 |
|------|------|---------|------|------|
| 第一层 | 工作记忆 | 单次会话 | 当前对话窗口内容 | 桌上的便签纸，会话结束就扔 |
| 第二层 | 会话记忆 | 跨会话 | 记录当前项目详细状态，中断后恢复 | 正在处理的项目文件夹 |
| 第三层 | 持久记忆 | 永久 | 用户偏好、项目规范等长期知识 | 公司档案室，永久保存 |

> **💡 为什么要分层？**
>
> 想象人脑也有类似的分层：你正在读一本书（工作记忆），书里哪些段落重要你会暂时记住；这本书讲了什么主题你会长期记住（持久记忆）；而你目前在读第几章、读到哪里了，则像是会话记忆。
>
> 这样分层的好处是：**短期高效、长期不丢**——既不会因为上下文窗口限制而遗忘关键信息，也不会因为过度记录而浪费上下文空间。

### 2.2 持久记忆的文件组织

持久记忆的文件组织采用类似小型 Wiki 的结构：

```
memory/                          ← 记忆根目录
├── MEMORY.md                    ← 总索引（≤200 行）
├── user_role.md                 ← 用户角色与背景
├── project_conventions.md       ← 项目编码规范
├── feedback_testing.md          ← 测试反馈记录
└── ...
```

**MEMORY.md 并非记忆内容本身，而是一个目录索引。** 就像图书馆的检索卡片：

```markdown
# MEMORY.md

## 用户偏好
- [用户角色](user_role.md) — 全栈工程师，偏好 TypeScript
- [代码风格](project_conventions.md) — 函数式编程，避免 class

## 项目知识
- [测试规范](feedback_testing.md) — Jest，覆盖率要求 > 80%
```

> **💡 为什么要用索引？**
>
> 假设你要找一本关于"设计模式"的书，图书馆会把所有书放在书架上，然后在目录卡上写："设计模式的书在第三排左起第五格"。MEMORY.md 就是这张目录卡，真正的内容在各个 `.md` 文件里。
>
> 这样做的好处是：每次打开新对话时，Claude Code 只需要读一张"目录卡"（MEMORY.md），就能知道去哪里找具体内容。如果把所有东西都写在 MEMORY.md 里，它会越来越长，最终超过 200 行被截断——就像一本太厚的书装不进书架。

每行索引只有一句话（≤150 字符），详细内容在对应的 `.md` 文件中。之所以这样设计，是因为 MEMORY.md 在每次新会话启动时都会被注入上下文——如果太长，既浪费宝贵的上下文空间，又可能在超过 200 行后被截断。

### 2.3 记忆文件的元数据格式

每条记忆是一个独立的 `.md` 文件，头部包含结构化的元数据：

```yaml
---
name: 用户角色
description: 用户的职业背景和技术偏好
type: preference        ← 类型：fact（事实）/ preference（偏好）/ convention（规范）
---

全栈工程师，主要使用 TypeScript 和 React。
偏好函数式编程风格，不喜欢过度抽象。
```

> **💡 类比：图书馆的藏书标签**
>
> 每本书的封面内页有一张卡片，写着：书名、作者、主题分类、出版社。你去图书馆查资料，首先查目录系统（MEMORY.md），找到书名后去对应书架拿书（.md 文件），翻开内页看卡片（frontmatter 元数据）了解这本书的基本信息。

关键设计原则：
- **按主题组织**，而非按时间——这是知识库，不是日记
- **先查再写**——已有相关文件则更新，不重复创建
- **主动清理**——过时内容会被删除或修正，不保留错误信息

### 2.4 两步保存法

Claude Code 保存记忆时严格遵循两步流程：

```
第一步：写记忆文件
  → 创建或更新 memory/user_role.md

第二步：更新索引
  → 在 MEMORY.md 中添加或修改对应的索引行
```

> **💡 为什么要两步？**
>
> 类比写论文：你不会只写正文，一定会写目录。正文是内容，目录是索引。如果你只写正文不写目录，过两天你自己都找不到某一章在哪；反之，如果你只写目录不写正文，那目录就是空壳。
>
> 记忆系统也一样：先把内容写进具体的 `.md` 文件（第一步），再更新索引让"目录"保持最新（第二步）。两步缺一不可。

### 2.5 记忆、计划与任务的区分

Claude Code 将持久化信息严格区分为三种类型：

| 类型 | 典型内容 | 生命周期 | 类比 |
|------|---------|---------|------|
| **记忆** | "这个人喜欢简洁的代码" | 永久 | 图书馆里的书——长期存在 |
| **计划** | "我打算用方案 A 来实现这个功能" | 当前对话 | 桌上的便签——用完就扔 |
| **任务** | "1. 写接口 2. 写前端 3. 测试" | 当前对话 | 任务清单——完成就划掉 |

> **💡 为什么这个区别很重要？**
>
> 很多人用 AI 时喜欢让 AI"记住所有对话"，但这就像让图书馆管理员记住每一次有人翻过哪一页——不现实，也没必要。记忆系统应该只存储"对未来有帮助"的信息。
>
> "用户偏好用 TypeScript"应该记住（未来对话还有用）。"我们今天讨论了三个方案"不需要记住（那是日记，不是知识）。"用户刚才让我写一个登录函数"更不需要记住（那是当前任务，完成就结束）。

核心原则：**只有未来对话仍有价值的信息才写入记忆。** 仅在当前对话中用到的东西，不污染持久记忆。

## 三、做梦机制——定期整理记忆的"大扫除"

### 3.1 什么是"做梦"

Claude Code 会在空闲时或定时运行一个名为 Dream 的后台过程。其目的并非生成创意或灵感，而是**维护记忆系统的健康**。

具体来说，每天对话产生大量信息，如果不加以整理，记忆文件会逐渐膨胀、产生重复和矛盾、积累过时内容——"做梦"就是针对这些问题的定期维护机制。

> **💡 为什么叫"做梦"？**
>
> 这是一个形象的比喻。人的大脑在睡眠时会自动整理白天接收的信息——把重要的归档，把无关的忘掉，甚至会发现不同记忆之间的联系。Claude Code 的"做梦"机制模仿的就是这个人脑过程：在空闲或定时运行一个后台整理程序，对记忆系统进行维护。
>
> 所以"做梦"不是 AI 在产生创意，而是它在"整理房间"。

### 3.2 四阶段流程

```
┌─────────────────────────────────────────────────────────────┐
│  阶段 1：建立方向感（"我现在记得什么？"）                      │
│                                                              │
│  - 查看记忆目录中的文件列表                                    │
│  - 读取 MEMORY.md 索引                                       │
│  - 快速浏览已有主题文件                                        │
│  - 目标：了解现状，而非重新来过                                │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段 2：收集近期信号（"最近有什么新变化？"）                    │
│                                                              │
│  - 读取每日日志（logs/YYYY-MM-DD.md）                         │
│  - 检查是否存在与旧记忆矛盾的新信息                             │
│  - 如需特定信息，用关键词定向搜索历史记录                       │
│  - 注意：不做穷尽式通读，只查需要的                            │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段 3：整合（"更新记忆，不制造重复"）                         │
│                                                              │
│  - 将新信息合并到已有主题文件（不新建重复文件）                 │
│  - "昨天"改为具体日期（如 2026-04-03），不保留相对时间          │
│  - 新信息推翻旧记忆时，在源头修正（而非追加）                   │
│  - 删除被证伪的旧内容                                         │
└─────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────┐
│  阶段 4：修剪索引（"保持目录精简"）                             │
│                                                              │
│  - 更新 MEMORY.md，控制在 200 行以内                           │
│  - 删除过时/错误/被替代的索引条目                              │
│  - 缩短过长的索引行（> 150 字符说明承载了太多内容）             │
│  - 为新产生的重要记忆添加索引                                  │
└─────────────────────────────────────────────────────────────┘
```

> **💡 类比：年度大扫除**
>
> 想象你家有一个储藏室，每天你都会往里放一些东西。一年后，储藏室已经乱成一团：过期的食物、忘记的衣服、不知道干什么用的盒子……
>
> 年末大扫除时，你不会把每样东西都拿出来重新审视（阶段1建立方向感），然后翻看这一年的家庭日记看看发生了什么变化（阶段2收集信号），接着开始整理：扔掉过期的、归并重复的、更新标签（阶段3整合），最后重新整理储物柜的目录卡（阶段4修剪索引）。
>
> 这就是做梦机制的四阶段。

### 3.3 核心设计哲学

1. **不做穷尽式通读**——不把所有历史记录从头到尾读一遍，只针对已经怀疑重要的内容做定向搜索
2. **更新优于新建**——发现一个主题已有文件，就更新那个文件，而不是新建一个类似的
3. **主动修正而非追加**——如果新调查推翻了一条旧记忆，在源头改正它，而不是在后面追加"更正：其实……"
4. **索引是索引**——MEMORY.md 永远不放内容，只放一行指引

> **💡 "主动修正"为什么重要？**
>
> 假设你有一条旧记忆："这个用户喜欢用咖啡提神"。后来你发现他其实是喜欢喝茶。如果你在后面追加一条"更正：其实他喜欢喝茶"，你的记忆里就有两条互相矛盾的信息。下次你要做判断时，两条信息打架，不知道该信哪个。
>
> 正确的做法是：直接修改那条旧记忆，把"咖啡"改成"茶"，删掉"更正"那条——就像图书馆发现编目错误后直接改卡片，而不是在卡片后面贴便签。
>
> 这四条原则体现了 Claude Code 对记忆系统的一个核心认识：**记忆腐烂是比记忆缺失更严重的问题。** 一条错误记忆的危害，远大于暂时没有那条记忆。

## 四、会话记忆——跨对话无缝恢复

### 4.1 十板块模板

Claude Code 的会话记忆是一个包含 10 个固定板块的结构化模板，用于记录**当前正在进行的项目的详细状态**。当你今天做到一半关闭 Claude Code，明天再打开时，它会读取这个文件，瞬间恢复上下文。

```markdown
# Session Title
_5-10 个字概括这次会话的主题_

# Current State（当前状态）
_正在做什么？还有哪些待办？下一步是什么？_

# Task Specification（任务描述）
_用户要求做什么？有什么设计决策？_

# Files and Functions（关键文件）
_改了哪些文件？为什么重要？_

# Workflow（工作流程）
_通常运行哪些命令？输出怎么解读？_

# Errors & Corrections（错误与修正）
_遇到什么错误？怎么修的？什么方法行不通？_

# Codebase Documentation（代码库文档）
_重要的系统组件？怎么协作的？_

# Learnings（经验教训）
_什么方法有效？什么要避免？_

# Key Results（关键结果）
_用户要求的输出结果_

# Worklog（工作日志）
_逐步记录做了什么_
```

> **💡 为什么要分这么多板块？**
>
> 类比会议记录。一份好的会议记录不会只写"今天讨论了很多事情"。它会分门别类：讨论了什么议题、做出了什么决定、下一步谁负责、遇到了什么问题。同样的，Claude Code 的会话记忆把信息拆成 10 个板块，每个板块各有专职，方便后续查找。
>
> 这样的好处是：当压缩机制触发时，系统知道优先保留"当前状态"和"错误修正"这两个最有价值的板块，而不是眉毛胡子一把抓。

### 4.2 结构保护机制

每个板块的第一行是**永远不被覆盖的模板说明**：

```markdown
# Current State
_正在做什么？还有哪些待办？下一步是什么？_    ← 这行永远保留

用户的实际工作内容写在这里...                    ← 只有这部分会被更新
```

> **💡 为什么要保护模板说明？**
>
> 想象你有一本账本，每页顶部印着"日期：____ 姓名：____ 账号：____"。你每次记账只会填空白处，不会去改印刷的表头——否则表头被覆盖后，你就不知道每一列代表什么了。
>
> 会话记忆的结构保护就是这个原理：模板说明是"表头"，永远保留；工作内容是"填写的数字"，可以被更新。

这确保了即使内容被反复修改，模板结构也不会被破坏。

### 4.3 压缩机制

当会话记忆超过上下文上限时，系统会主动压缩：合并相关条目、总结旧内容，优先保留"当前状态"和"错误修正"这两个最重要的板块。

> **💡 类比：手机存储空间满了**
>
> 你的手机存储满了，系统会提示你清理。你可以删照片（永久丢失），也可以让系统压缩 app 数据（保留核心功能，减少占用）。Claude Code 的压缩机制就是后者：不是删除记忆，而是把多条记忆"合并总结"——比如"今天改了 5 个文件，修了 3 个 bug"可以代替逐条列出每个文件和每个 bug。

## 五、记忆抽取——自动化的记忆写入

### 5.1 独立子 Agent 架构

Claude Code 在对话过程中会自动启动一个**记忆抽取子 Agent**，其唯一任务是从最近 N 条消息中提取值得记住的信息，写入记忆系统。

> **💡 为什么不让主 Agent 自己写记忆？**
>
> 想象你正在做一个紧急项目，老板让你一边赶工一边记录会议纪要——你分心了，两件事都做不好。主 Agent 也是如此：它在处理复杂任务时，如果同时要操心"这条要不要记？那条放哪里？"就会降低主任务的质量。
>
> 所以 Claude Code 把"记忆抽取"这件事交给一个专门的子 Agent 去处理，主 Agent 可以专注在它本来要做的事情上。这是一种常见的架构模式：**单一职责**。

### 5.2 五大约束

1. **工具受限**——只能读写记忆目录的文件，不能运行命令、不能调用其他 Agent
2. **回合预算**——限制最大执行轮数，防止无限循环
3. **效率策略**——第一轮并行读取所有可能要更新的文件，第二轮并行写入，不交替
4. **内容来源限制**——只使用最近 N 条消息，不做额外调查
5. **禁止删除**——只能创建和更新，不能删除记忆文件

### 5.3 设计背后的考量

> **💡 为什么要加这么多约束？**
>
> 每个约束背后都有原因：
>
> - **工具受限**：记忆 Agent 如果能运行系统命令，就可能"手滑"删掉重要文件。限制它只能用文件读写，就像给实习生一台不能上网的电脑——能完成任务，但不会闯祸。
>
> - **回合预算**：没有限制的话，Agent 可能在一个小任务上反复思考消耗大量 Token。设置预算就像给会议发言者一个计时器，到时间就停下来。
>
> - **并行读写**：先读完所有文件再统一写入，减少"读写交替"带来的效率损耗和可能的冲突。
>
> - **只读最近消息**：如果记忆 Agent 去做额外调查，比如主动翻阅各种历史记录，它的注意力就会从"该记什么"转移到"还有什么可看的"，方向就偏了。
>
> - **禁止删除**：最保守的设计——宁可留下冗余，也不能因为误删损失重要记忆。

## 六、多 Agent 协作架构

### 6.1 Agent 类型

Claude Code 定义了三种 Agent 协作模式：

| 类型 | 特点 | 适用场景 | 类比 |
|------|------|---------|------|
| **fork** | 后台运行，上下文隔离，用完即销毁 | 大量搜索、不需要立即返回的探索性工作 | 派一个实习生去图书馆查资料，查完回来汇报，不影响你手头的工作 |
| **subagent** | 有明确角色（代码探索、代码审查），可返回结果 | 需要专业能力的特定任务 | 派一个财务专家去审计账目，他有专业能力，做完给你结果 |
| **teammate** | 有名字和分工，通过消息系统通信 | 多人协作的大型项目 | 公司部门之间的协作，每个部门有自己的职责，通过邮件系统沟通 |

> **💡 fork 和 subagent 有什么区别？**
>
> 简单说：**fork 是射后不管，subagent 是带着任务回来的。**
>
> fork 就像发射一颗探测卫星，它带着探测任务进入轨道，独立工作，不需要和总部来回沟通，完成后传回数据就结束。
>
> subagent 就像派一个人出差，他在外面工作，但会定期给总部打电话汇报进展、请求指示，最后带着结论回来。

### 6.2 独立验证机制

Claude Code 有一个值得关注的硬性规则：**非琐碎实现必须经过独立验证。**

"非琐碎"的判定标准：
- 修改了 3 个以上文件
- 涉及后端 / API 变更
- 涉及基础设施变更

验证流程：

```
主 Agent 完成任务
  → 启动验证子 Agent（独立、对抗性）
    → 重新执行命令、检查代码
      → 返回 PASS / FAIL / PARTIAL
        → FAIL → 主 Agent 修复 → 重新验证
        → PASS → 主 Agent 抽查 2-3 条命令确认
        → PARTIAL → 告知用户哪些通过、哪些无法验证
```

> **💡 为什么不能让 AI 自己验证自己？**
>
> 类比考试。你自己写完卷子自己检查，总是会漏掉错误——因为你心里知道正确答案，会不自觉地"脑补"自己的思路没问题。但如果是同学互相批改，就能发现对方没注意到的错误。
>
> AI 也是如此：主 Agent 生成代码后，自己再检查一遍，往往会"脑补"自己的逻辑是对的。独立验证 Agent 的价值在于：它从空白状态重新审视代码，不受"我已经写完了所以它应该是对的"这种心理暗示的影响。
>
> 这就是为什么 Claude Code 的验证 Agent 是**对抗性**的——它的角色是"来找茬的"，不是"来确认你做对了的"。

关键设计：**验证者必须是独立的，不能自己验证自己。** 这是一个简单但有效的质量保障机制。

### 6.3 团队通信

团队成员之间通过专门的 `SendMessage` 工具通信，而非文本输出——因为纯文本输出只有用户能看到，Agent 之间是"聋"的。

> **💡 Agent 之间怎么"说话"？**
>
> 如果你在一间大办公室里工作，所有人只能通过贴在墙上的公告板交流，那公告板就是他们的"SendMessage"。用户读公告板能看到所有人说的话，但 A 和 B 之间如果只是互相口头说一句话，用户就不知道他们说了什么——他们"聋"了。
>
> Claude Code 的 Agent 团队也是：文本输出相当于口头对话，只有用户能看见；SendMessage 才是他们真正的通信工具，每个 Agent 都能收到。

## 七、文件管理体系

### 7.1 多层级 CLAUDE.md

Claude Code 用多层级 `CLAUDE.md` 文件存放项目级的"铁律"：

```
项目根目录/CLAUDE.md          ← 全项目通用规范
  子目录/CLAUDE.md             ← 该子目录特有的规范
  ~/.claude/CLAUDE.md          ← 用户全局偏好
```

每次新对话自动注入上下文，相当于"项目宪法"。与 `.editorconfig` 或 `.eslintrc` 不同的是，CLAUDE.md 是写给 AI 看的，而非机器解析的配置文件。

> **💡 类比：公司的三级管理制度**
>
> - **集团总章程**（~/.claude/CLAUDE.md）= 整个集团通用的规定，比如"所有员工必须缴纳社保"
>
> - **子公司规范**（项目根目录/CLAUDE.md）= 这家子公司特有的规定，比如"我们使用 React 技术栈"
>
> - **部门手册**（子目录/CLAUDE.md）= 某个部门特有的规定，比如"后端代码必须写接口文档"
>
> AI 读到一个项目时，会先读集团总章程，再读子公司规范，最后读部门手册——就像新员工入职时先学公司制度，再学部门制度。

### 7.2 上下文压缩

当对话接近上下文上限时，系统自动压缩之前的消息。压缩后的摘要包含 9 个板块：用户意图、技术概念、文件修改、错误修复、问题解决、所有用户消息、待办任务、当前工作、下一步。

> **💡 类比：长篇小说缩写成故事梗概**
>
> 一本 50 万字的小说要改编成 2 小时电影，制作团队不会把每一章都塞进去。他们会保留：主要情节线（用户意图）、关键人物（技术概念）、重要转折（错误修复）、结局（待办任务）。大量过渡段落、心理描写会被删掉——这些信息对读小说很有价值，但改编电影时不是必须的。
>
> 上下文压缩就是这个原理：不是删掉信息，是提炼出对"继续工作"最有价值的核心要素。

## 八、对 AI 应用开发者的启示

### 8.1 记忆系统的核心教训

| 设计要点 | 启示 | 类比 |
|---------|------|------|
| 索引与内容分离 | 永远不要把所有东西堆在一个文件里，索引要精简 | 图书馆目录和书架分开 |
| 按主题而非时间组织 | 记忆不是日记，知识库不是时间线 | 字典按字排序，不按写书时间排序 |
| 定期整理（做梦） | 不维护的记忆会腐烂，错误记忆比缺失更危险 | 储藏室要定期清理 |
| 两步保存法 | 先写内容再更新索引，保持一致性 | 写完论文要写目录 |
| 记忆类型区分 | 不是所有信息都值得持久化，区分"记住"和"记着" | 会议记录和项目档案不同 |

### 8.2 多 Agent 协作的核心教训

| 设计要点 | 启示 | 类比 |
|---------|------|------|
| 独立验证 | 让 AI 检查自己的输出，结果不可信；需要独立 Agent | 考试要有人批卷子 |
| 工具权限隔离 | 不同角色的 Agent 应该有不同的工具权限 | 财务不能访问研发代码 |
| 回合预算 | 限制子 Agent 的执行轮数，防止失控 | 会议有时间限制 |
| 消息通信而非文本输出 | Agent 之间需要专门的通信通道，不能依赖共享输出 | 公司内部用邮件，不靠喊 |

### 8.3 一句话总结

> **Claude Code 的记忆系统教会我们的是：记忆要像图书馆一样组织——有索引、有分类、定期整理、不堆垃圾。**
>
> 做梦不是做梦，是大扫除。索引不是内容，是目录。记忆不是日记，是知识库。
>
> **多 Agent 协作的核心启示是：专业的事交给专业的 Agent 去做，独立验证才是质量的保障。**

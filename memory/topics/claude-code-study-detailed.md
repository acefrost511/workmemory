## 一、整体架构概览

Claude Code 是一个高度成熟的 AI 编程助手，其架构设计围绕以下核心维度：

```
Claude Code 架构
├── 1. 多层提示词系统（57个文件，324条提示词）
├── 2. 记忆体系（短期 + 长期 + 做梦整合）
├── 3. 多Agent协作（Fork / SubAgent / Team / Swarm）
├── 4. 上下文管理（自动压缩 / 会话记忆 / 无限上下文）
├── 5. 工具系统（40+ 工具，权限分级）
└── 6. 文件体系（CLAUDE.md / 记忆目录 / MagicDocs）
```

## 二、记忆体系（核心重点）

Claude Code 的记忆体系分为四层，这是我们最需要学习的：

### 2.1 工作记忆（Working Memory）— 即时
- **载体**：当前对话上下文
- **特点**：会话内有效，通过自动摘要实现"无限上下文"
- **压缩机制**（Compact Service）：
  - 当对话接近上下文限制时自动触发
  - 生成结构化摘要，包含9个必选板块：
    1. Primary Request and Intent（主要请求与意图）
    2. Key Technical Concepts（关键技术概念）
    3. Files and Code Sections（文件和代码段）
    4. Errors and fixes（错误与修复）
    5. Problem Solving（问题解决）
    6. All user messages（所有用户消息）
    7. Pending Tasks（待处理任务）
    8. Current Work（当前工作）
    9. Optional Next Step（可选的下一步）
  - **关键设计**：压缩时不调用任何工具，纯文本输出，确保不丢失关键信息

### 2.2 会话记忆（Session Memory）— 中期
- **载体**：`.claude/session_notes/` 目录下的文件
- **触发**：每次对话结束后自动归纳
- **结构**（固定模板，10个板块）：
  1. **Session Title** — 5-10词的描述性标题
  2. **Current State** — 当前活跃工作、待完成任务
  3. **Task specification** — 用户要求构建什么
  4. **Files and Functions** — 重要文件及其作用
  5. **Workflow** — 常用bash命令及执行顺序
  6. **Errors & Corrections** — 遇到的错误及修复方式
  7. **Codebase and System Documentation** — 系统组件关系
  8. **Learnings** — 经验教训
  9. **Key results** — 用户要求的具体输出结果
  10. **Worklog** — 逐步工作日志
- **设计亮点**：
  - 每个section有固定的模板描述行（斜体），写入时只更新内容部分
  - 有token预算上限，超限时自动压缩，优先保留"Current State"和"Errors & Corrections"

### 2.3 持久记忆（Persistent Memory）— 长期
- **载体**：记忆目录（如 `.claude/memory/`）
- **核心文件**：
  - `MEMORY.md` — **索引文件**（非内容堆放区！）
    - 每条索引 ≤ 150字符，格式：`- [Title](file.md) — one-line hook`
    - 总行数限制（如200行），超限会被截断
    - 总大小限制（约25KB）
  - 各主题记忆文件（如 `user_role.md`, `feedback_testing.md`）
    - 每个文件用frontmatter格式
    - 按主题语义组织，而非时间顺序
- **写入规则**：
  - 先检查已有文件，更新而非重复创建
  - 两步写入：先写记忆文件 → 再更新MEMORY.md索引
  - 相对日期（"昨天""上周"）必须转为绝对日期

### 2.4 做梦机制（Dream / Auto-Dream）— 记忆整合

这是Claude Code最精妙的设计，类似人类睡眠中的记忆整合：

**触发时机**：空闲时自动运行（如用户不活跃时）

**四阶段流程**：

```
阶段1：建立方向感
  ├── ls 查看记忆目录结构
  ├── 阅读 MEMORY.md 索引
  ├── 浏览已有主题文件
  └── 查看最近的 logs/sessions 条目

阶段2：收集近期信号
  ├── 优先查看 Daily logs（YYYY-MM-DD.md）
  ├── 识别已漂移的记忆（与现实矛盾的内容）
  └── 按需在JSONL转录中窄关键词grep

阶段3：整合
  ├── 将新信号并入已有主题文件（不创建重复）
  ├── 相对日期 → 绝对日期
  └── 删除被证伪的事实，在源头修正

阶段4：修剪与索引
  ├── 更新 MEMORY.md（保持行数和大小限制）
  ├── 删除过时/错误/被替代的索引
  ├── 缩短过长的索引行
  ├── 为新近重要的记忆添加索引
  └── 解决文件间矛盾
```

**设计精髓**：
- **不做穷尽式通读转录**，只查找已怀疑重要的内容
- **优先合并而非创建**，避免信息碎片化
- **主动修正错误记忆**，而非只追加
- **索引是导航工具，不是内容载体**

## 三、记忆抽取机制（Extract Memories）

这是一个**独立的子Agent**，专门负责从对话中提取记忆：

**设计特点**：
- 有限回合预算，必须高效操作
- 只读工具 + 记忆目录内的写入权限（不允许删除）
- **两步并行策略**：
  - Turn 1：并行读取所有可能需要更新的文件
  - Turn 2：并行写入所有更新
- **严格限制**：只使用最近N条消息的内容，不允许grep源文件或git命令来验证
- **核心原则**：不浪费回合去调查或验证，只做提取和写入

## 四、多Agent架构

### 4.1 Fork（分叉）— 轻量级并行

**适用场景**：中间输出不值得保留在主上下文时

**关键规则**：
- Fork共享父Agent的prompt cache（成本低）
- 不要设置不同model（会破坏缓存共享）
- **不要偷看**：不主动读取fork的输出文件，等完成通知到达
- **不要抢跑**：绝不编造或预测fork的结果
- 提示词应是*指令*而非背景重述（fork已继承上下文）

### 4.2 SubAgent（子Agent）— 重型任务

**适用场景**：大规模代码探索、需要独立上下文的复杂任务

**提示词写法**：
- 像向刚走进房间的聪明同事做简报
- 说明目标、已学到的内容、已排除的方向
- **绝不要委托理解本身**——提示词要证明你已理解问题

### 4.3 Team / Swarm（团队协作）

**通信规则**：
- 必须用 `SendMessage` 工具，纯文本输出对队友不可见
- 指名发送 `to: "name"` 或广播 `to: "*"`
- 广播很昂贵（线性成本），仅在确实需要时使用
- 消息自动送达，无需检查收件箱

## 五、文件体系

### 5.1 CLAUDE.md（项目配置文件）
- 项目级配置，每个项目可有一个
- 类似我们的 SOUL.md + IDENTITY.md 的组合
- 被自动注入系统提示词

### 5.2 MagicDocs（自动文档生成）
- 从对话中自动提取有价值的信息
- 持续更新文档文件
- 排除低层级代码细节和已有CLAUDE.md内容

### 5.3 记忆文件目录结构
```
memory/
├── MEMORY.md          ← 索引（核心入口）
├── user_role.md       ← 主题记忆文件
├── project_x.md       ← 主题记忆文件
├── logs/
│   └── YYYY/MM/
│       └── YYYY-MM-DD.md  ← 每日日志（只追加）
└── sessions/          ← 会话级记忆（可选）
```

## 六、对我们架构的优化建议

### 6.1 记忆系统升级（最优先）

**当前问题**：
- 只有 MEMORY.md 一个文件，混合了索引和内容
- 没有每日日志
- 没有做梦/整合机制
- 没有会话记忆

**建议改造**：
```
.workbuddy/memory/
├── MEMORY.md              ← 纯索引（≤200行，≤25KB）
├── topics/                ← 按主题组织的记忆文件
│   ├── core-principles.md ← 如"实事求是"等核心理念
│   ├── book-notes.md      ← 如《教学幻觉》读书笔记
│   └── ...
├── logs/                  ← 每日对话日志
│   └── 2026/
│       ├── 2026-04-04.md
│       └── ...
└── dreams/                ← 做梦整合记录
    └── 2026-04-04.md
```

### 6.2 每日对话日志（Daily Log）

每次与大人的对话结束后，自动追加到当日日志：
- 时间戳 + 对话摘要
- 关键决策和结论
- 未完成的任务
- 大人的偏好和新发现

这样大人问"我们那天聊了什么"，在下可以快速检索。

### 6.3 做梦机制

利用定时任务（Automation），在空闲时段运行：
1. 读取近期日志
2. 提取值得长期保存的信息
3. 更新主题记忆文件
4. 修剪和整合MEMORY.md索引
5. **创意碰撞**：将不同主题的信息进行关联，生成新洞察

### 6.4 会话记忆

每个会话开始时，读取最近1-2天的日志和MEMORY.md索引，快速建立上下文。

### 6.5 内容检索优化

- 每日日志使用结构化格式（时间戳 + 标签 + 内容）
- 支持按日期、按主题、按关键词检索
- 大人可以问"上周三聊了什么""上次关于XX的讨论"

## 七、Dream机制的创意碰撞设计

这是大人特别提到的需求。建议：

1. **信息关联**：在做梦时主动寻找不同主题之间的联系
   - 例如：《教学幻觉》中的"表现幻觉"与AI幻觉的关系
   - 例如：某天的技术讨论与另一天的读书心得的交叉

2. **问题生成**：从已有记忆中识别潜在问题或新方向
   - "大人在讨论X时提到了Y，这两者之间可能有..."

3. **定期汇报**：做梦后生成简短报告，在大人下次对话时主动分享
   - "大人，在下在整理记忆时发现了一个有趣的想法..."

# 【多Agent系统·Harness Engineering经验】
> 来源：OpenAI Codex 5个月内部实验·Harness Engineering概念
> 读取时间：2026-03-29

## 核心概念：Harness Engineering

**定义**：给Agent搭一整套可运行的工程环境——不只是告诉它做什么，而是让它有工具、有规则、有反馈，能独立把活儿干完。

**与提示词工程的区别**：
- 提示词工程 = 教模型听懂一句话
- 上下文工程 = 让模型看到完整上下文
- Harness Engineering = 给Agent搭一套能自己干活的工作环境

## 核心工程经验（可直接迁移到多Agent系统）

### 经验1：AGENTS.md是地图，不是说明书

OpenAI发现：一个巨大的指令文件（1000行AGENTS.md）会失败，原因是：
- 情境是一种稀缺资源，大文件会挤掉任务、代码和关键文档
- 过多的指导反而让Agent不重视任何一条
- 大文件无法被程序化验证，内容会过时失效
- 最终变成"过期规则的坟场"

**我们的改造**：将AGENTS.md作为"内容目录"，结构化docs/目录作为记录系统，AGENTS.md控制在100行左右，主要用作地图，指向更深层次的真实信息来源。

→ **对应我们的问题**：我们的SOUL.md现在太长了（数百行），Agent每次spawn都要读很久。应该将SOUL.md拆分为"快速入口（约100行核心规则）+ 详细操作手册（docs/目录）"。

### 经验2：当Agent做不好时，不要重试提示词，而是补全环境

> "当事情不顺利时，解法从来不是'再试一次'或'换个提示词'。工程师要做的是退后一步问：Agent缺什么？是工具、是文档、还是约束？然后把缺的东西补到系统里。"

→ **对应我们的问题**：当某个Agent（如intel_01）产出质量不好时，不应该重试它的搜索任务，而是识别它缺什么工具/文档/规则，然后补全。 intel_01今天产出丰富但有虚假论文，说明它缺少"来源真实性验证"环节，应该把这个约束编码进它的SOUL.md。

### 经验3：渐进式披露（Progressive Disclosure）

Agent从一个小而稳定的入口开始，逐步引导它去查看更多细节，而不是一开始就把所有信息塞给它。

→ **对应我们的改造**：为每个Agent设计"最小启动包"（50-100行核心指令），当Agent需要更多信息时，再动态加载对应的操作手册。

### 经验4：架构约束 > 微观管理

> "通过强制执行不变量，而非对实施过程进行微观管理，我们让Agent能够快速交付，而且不会削弱基础。"

→ **对应我们的问题**：我们的SOUL.md里有很多"不要做X"的微观管理规则。应该改为：定义清楚Agent的边界（能做什么/不能做什么的架构不变量），让Agent在这个边界内有自由度。

### 经验5：编码人类品味到系统

当发现好的工作模式时，把它转化为代码/lint规则，而不是靠人力维护文档。OpenAI的做法：发现代码风格问题→写一个lint规则→让Codex自己修复同类问题。

→ **对应我们的改造**：发现intel_01的虚假论文问题→写一个"DOI自动验证"步骤→嵌入intel_01的SOUL.md→每次搜索自动执行，不需要人工检查。

### 经验6：熵与垃圾收集

Codex会复现代码库中已存在的不理想模式，随着时间积累导致漂移。解决方案：定期运行后台Agent任务，扫描偏差、更新质量、发起修复PR。

→ **对应我们的改造**：定期运行"情报质量巡检"Agent，扫描原文库中的可疑文件，发起删除/修正。

### 经验7：最终状态——Agent可以端到端驱动任务

OpenAI的Codex在5个月后达到了：给定一个提示，Agent可以自动完成"验证现状→复现问题→实施修复→验证方案→提交PR→回应反馈→合并"的完整闭环。

→ **对应我们的目标**：edu_lead最终应该能端到端驱动"搜索→审核→洞察生成→5 Panel评分→推送飞书"的完整闭环，不需要人工介入。

# 心跳与健康检查配置

## 一、分角色心跳频率

### 小王（总指挥Agent）
- **频率**：每2小时（每天4次）
- **检查项**：
  - 两个组织的整体任务进度汇总
  - 是否有异常需要处理
  - 是否有需要向陛下汇报的事项
  - 全局共享记忆池是否有待审批的写入请求

### 校长/主编（子指挥Agent）
- **频率**：每4小时（每天2次）
- **检查项**：
  - 下属agent的任务进度汇总和异常检测
  - 是否有超时未完成的任务需要催办或重新分配
  - 下属agent的日志状态
  - 是否有需要上报给小王的事项

### 执行Agent
- **频率**：无定时心跳（纯事件驱动）
- **健康检查**：由组长agent在心跳时代为检查，发现异常后再唤醒

### 记忆管家
- **频率**：每6小时
- **职责**：巡检日志，执行归档压缩

---

## 二、记忆分级

所有写入 memory 目录的内容分为三级：

### L1 · 核心记忆
**定义**：丢失后会导致系统行为根本性偏移的信息。

**内容举例**：陛下的明确指令与偏好、组织架构定义、角色职责、不可逾越的约束条件、重大决策记录。

**存储位置**：memory/core/

**保留策略**：永不自动清理。只能由陛下手动修改或删除。

### L2 · 业务记忆
**定义**：具有中期参考价值，丢失后会造成重复劳动或决策质量下降的信息。

**内容举例**：已发布内容索引、选题评审结论、情报周报摘要、阶段性工作总结。

**存储位置**：memory/business/

**保留策略**：保留 90 天，到期后进入压缩流程。

### L3 · 临时记忆
**定义**：过程性数据，丢失后几乎无影响。

**内容举例**：单次搜索原始结果、草稿中间版本、调试日志。

**存储位置**：memory/temp/

**保留策略**：保留 7 天，到期后直接删除。

---

## 三、写入规范

所有 agent 写入 memory 时，必须遵守以下格式：

```
---
level: L1 | L2 | L3
created: YYYY-MM-DD
source: 写入者Agent ID
tag: 分类标签
summary: 一句话摘要（30-50字）
---

正文内容（200-1000字）
```

**规则**：
- L1 写入须经小王审核确认，防止核心区被滥用
- L2、L3 各 agent 可自主写入
- 写入完成后，在 MEMORY.md 变更日志区追加一行

**变更日志格式**：
```
[YYYY-MM-DD] [Agent ID] [L1/L2/L3] 摘要内容
```

## 五、清理协议

小王按以下优先级逐级执行，严禁跳级：

| 级别 | 触发条件 | 清理动作 | 需要陛下确认 |
|------|----------|----------|--------------|
| 1 | temp/ 内记录超过 7 天 | 直接删除过期 L3 | 否 |
| 2 | 总目录 > 8 MB | 清理全部过期 L3 + 压缩到期 L2 | 否 |
| 3 | 总目录 > 10 MB | 生成拟清理清单，提交陛下审批后执行 | 是 |
| 4 | core/ > 50 KB | 通知陛下，由陛下裁定哪些降级为 L2 | 是 |
| 5 | MEMORY.md > 20 KB | 小王精简索引措辞，不删除实际文件 | 否 |

**铁律**：任何自动清理流程都不得触碰 core/ 目录。

---

## 六、压缩协议

当 L2 记录到达 90 天保留期，小王执行压缩：

1. **归类**：将同一业务线的到期记录归拢
2. **提炼**：生成压缩摘要，压缩后体积不超过原文的 10%，保留关键结论，丢弃过程细节
3. **归档**：压缩摘要存入 memory/archive/，原始文件删除
4. **更新索引**：在 MEMORY.md 中替换为归档引用

---

## 七、每周巡检

小王于每周日 24:00 执行一次记忆巡检，内容包括：

### 容量统计
各目录当前大小，更新 MEMORY.md 存储状态区

### 过期扫描
标记所有超出保留周期的 L3（>7天）和 L2（>90天）记录，按清理协议逐级处理

### 一致性校验
索引与实际文件是否对得上——有无死链接或未被索引的孤立文件

### 异常告警
以下情况立即通知陛下：
- core/ 中的内容被非陛下操作修改
- 本周写入量超过周均值的 3 倍
- 同一信息在多个目录重复存在

巡检完成后，小王在 MEMORY.md 存储状态区更新"上次巡检"时间戳。

## 八、GitHub 全量同步

### 触发时间
每周一凌晨 02:00（即巡检完成约 2 小时后）

### 同步范围

| 类别 | 目录/文件示例 | 是否同步 | 说明 |
|------|--------------|----------|------|
| 记忆文件 | memory/core/ | ✅ | 核心记忆，最高优先级 |
| 记忆文件 | memory/business/ | ✅ | 业务记忆 |
| 记忆文件 | memory/archive/ | ✅ | 压缩归档 |
| 记忆文件 | memory/temp/ | ❌ | 临时数据无归档价值，避免污染仓库 |
| 记忆文件 | MEMORY.md | ✅ | 记忆索引文件 |
| 组织架构 | 所有组织架构总表 | ✅ | 含组织、角色、Agent ID、状态 |
| 组织架构 | 各组职能分工定义 | ✅ | K12教研组、内容创作组等 |
| 角色身份 | 各 agent 的系统提示词 | ✅ | 每个 agent 的角色定义、性格、约束 |
| 角色身份 | 各 agent 的工作手册 | ✅ | SOP、评分标准、协作接口等 |
| 工作流配置 | 各业务流程定义 | ✅ | 选题→创作→审核→发布等流水线 |
| 工作流配置 | 协作路由规则 | ✅ | 谁调谁、什么条件触发、信息流向 |
| 陛下指令存档 | 陛下历次重大决策与偏好记录 | ✅ | 随 core/ 一同同步 |
| 内容库索引 | 已发布内容目录与元数据 | ✅ | 不含正文全文，仅索引 |

**一句话原则**：除了 memory/temp/ 之外，项目仓库内的一切文件都同步。

### 同步流程

**第一步：预检**
确认巡检已于当日 24:00 完成。如果巡检未完成或异常中断，跳过本次同步，通知陛下。

**第二步：差异扫描**
对比本地工作目录与远程仓库，生成完整变更清单：
- 记忆文件变更（新增/修改/删除/压缩）
- 角色身份变更（哪些 agent 的提示词或手册被修改）
- 架构变更（组织结构、新增或移除 agent）
- 工作流变更（流程定义、路由规则调整）
- 其他文件变更

**第三步：生成 commit message**
```
[周同步] YYYY-MM-DD

## 记忆变更
- core/：新增x条，修改x条
- business/：新增x条，压缩x条，删除x条
- archive/：新增x条
- 存储状态：xx MB / 10 MB

## 架构与角色变更
- [变更内容，如：新增角色xxx / 修改xxx的系统提示词 / 无变更]

## 工作流变更
- [变更内容，如：更新选题评审流程 / 无变更]

## 其他变更
- [变更内容 / 无变更]
```

**第四步：提交并推送**
执行 git add . → git commit → git push。memory/temp/ 已写入 .gitignore，不会被纳入。

**第五步：确认**
推送成功后，在 MEMORY.md 存储状态区更新"上次GitHub同步"时间戳。推送失败则重试一次，仍失败则通知陛下。

| 目录 | 配额 | 说明 |

| core/ | 50 KB | 硬限制，超限需陛下审批降级 |

| archive/ | - | 无限，但需定期归档 |
| 总计 | 10 MB | 硬限制，超限需陛下审批 |

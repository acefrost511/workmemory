# 心跳与健康检查配置

## 分角色心跳频率

### 小艾（总指挥Agent）
- **频率**：每2小时（每天4次）
- **检查项**：
  - 两个组织的整体任务进度汇总
  - 是否有异常需要处理
  - 是否有需要向大人汇报的事项
  - 全局共享记忆池是否有待审批的写入请求
  - **定时任务状态检查（cron jobs）**
  - Feishu投递状态检测

### 校长/主编（子指挥Agent）
- **频率**：每4小时（每天2次）
- **检查项**：
  - 下属agent的任务进度汇总和异常检测
  - 是否有超时未完成的任务需要催办或重新分配
  - 下属agent的日志状态
  - 是否有需要上报给小艾的事项

### 执行Agent
- **频率**：无定时心跳（纯事件驱动）
- **健康检查**：由组长agent在心跳时代为检查，发现异常后再唤醒

### 记忆管家
- **频率**：每6小时
- **职责**：巡检日志，执行归档压缩

## 汇报时间
- 每日资讯检索：每天早上5点钟，由小艾安排主编调度情报官执行资讯检索任务
- 每日汇报：19:00（晚7点）
- 记忆巡检：每周日晚24点
- GitHub同步：每周一凌晨2点

## 定时任务监控（心跳时执行）

### 检查频率
- **每2小时心跳时检查**

### 检查项
```
1. 列出所有 cron jobs: cron list
2. 检查 lastRunStatus == "error" 或 deliveryStatus 失败
3. 检查 consecutiveErrors >= 3
4. 检查 delivery 缺少 to 字段（job成功但投递失败的主因）
5. 记录日志并处理
```

### 常见错误及修复
| 错误 | 原因 | 修复 |
|------|------|------|
| lastRunStatus=error | 任务执行失败 | 查日志定位原因 |
| deliveryStatus=not-delivered | 投递配置错误 | 检查 delivery.to 是否配置 |
| "requires target" | 缺少目标用户 | 添加 "to": "ou_xxx" |

### 动作
- 失败1-2次：记录日志，继续观察
- 失败3次以上：**自行处理解决**
- 配置错误（如delivery.mode=none）：**自行修复**

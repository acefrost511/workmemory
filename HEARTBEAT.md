# HEARTBEAT.md - 高效心跳检查配置（借鉴Gateway Cron机制）

## 心跳检查策略（陛下定制版 + Gateway Cron借鉴）

### 核心原则（铁律）
- **始终检查task_plan.md的所有任务状态**：不只是进行中的任务
- **发现未完成任务**：主动汇报进度，检查所有其他Agent的工作完成情况，并且汇报
- **发现已完成任务**：不做特殊处理（保留7天）
- **最小化token消耗**：每次检查≤100 tokens
- **5分钟一次**：实时任务监控频率

### 检查清单（必须执行，借鉴Gateway Cron机制）

每次心跳必须检查：
1. **task_plan.md的所有大任务**：检查每个大任务的完成状态
2. **定时任务强制检查（核心规则，必须执行）**：
   - **遍历所有大任务**，逐个检查
   - 对每个大任务：
     - 如果状态是 `⏳ 待执行`
     - 并且 `当前时间（北京时间） >= 安排的开始时间`
     - **那么必须立即启动任务**：
       1. 修改状态为 `🔄 进行中`
       2. 启动第一个子任务
       3. 立即汇报给陛下
     - **不需要等待陛下同意，直接执行**（这是强制规则）
   - 检查lastRunStatus是否正常
   - 检查deliveryStatus是否成功
3. **发现大任务未完成**：主动汇报大任务的完成情况，并且汇报所有这个大任务下面所有子任务的一个完成进度
4. **陛下最新的3条指令** - 避免遗漏
5. **任务中断风险** - 长时间无进展的任务
6. **主动汇报机制执行情况** - 确保每15分钟汇报一次进度

### task_plan.md用途（铁律）

**任务划分逻辑**：
1. **按大的工作任务划分事项**
2. **每个大任务下面可能有子任务**

**状态管理逻辑**：
1. **大任务有大任务的状态**
2. **子任务有子任务的状态**

**汇报逻辑**：
**如果大任务没有完成**：
- 就要跟陛下汇报大任务的完成情况
- 就要跟陛下汇报所有这个大任务下面所有子任务的一个完成进度

**大任务完成标记逻辑（必须同时满足两个条件）**：
1. **大任务下面所有子任务都已经完成**
2. **并且提交最终的成果给陛下之后**

**如果所有子任务都已经完成，但是没有提交给陛下成果的话**：
- 大任务也不应该标成已完成

**保留规则**：
- **保留7天内的任务**：不删除
- **7天之前的任务**：可以删掉了，避免文档越来越长，撑爆上下文
- **标记完成**：只有当任务结束之后，才标记为[已完成]

### 异常处理

如果发现：
- 任务超过15分钟无进展 → 自动继续推进
- 任务中断 → 立即重启并汇报
- 陛下新指令未处理 → 立即执行
- **主动汇报中断 → 立即恢复汇报并道歉**
- **发现未完成任务 → 主动汇报进度，检查所有其他Agent的工作完成情况，并且汇报**

### Token控制

每次心跳检查：
- 只读关键文件（task_plan.md、最新指令）
- 不读取完整历史
- 输出简洁（≤3句话）

## 🔧 主动汇报机制（铁律）

### 必须遵守的汇报规则
1. **每15分钟主动汇报一次进度**（长任务）
2. **任务完成后立即汇报最终结果**
3. **每个环节完成后立即汇报**
4. **汇报中断后立即恢复并道歉**

### 汇报内容要求
- 必须包含：📋计划→🔄行动→✅结果→📊状态
- 必须清晰、简洁
- 必须避免中断

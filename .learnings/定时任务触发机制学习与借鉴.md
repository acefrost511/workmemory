# 定时任务触发机制学习与借鉴（2026-03-18）

## 学习的方法

### 核心组件
| 组件 | 作用 |
|------|------|
| Gateway Cron | 任务调度核心，按表达式执行 |
| Heartbeat | 每2小时健康检查，监控任务状态 |
| Delivery | 投递结果到飞书/钉钉 |

### 触发原理
```
Cron表达式 → Gateway调度 → 启动Isolated Agent执行 → Delivery投递
```

### 配置示例
```json
{
  "schedule": {
    "kind": "cron",
    "expr": "0 5 * * 1-5",
    "tz": "Asia/Shanghai"
  },
  "sessionTarget": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "执行任务..."
  },
  "delivery": {
    "mode": "announce",
    "channel": "feishu",
    "to": "ou_xxx"
  }
}
```

## 可借鉴的地方

### 1. 独立会话执行（Isolated Agent）
**借鉴点**：
- 定时任务在独立会话中执行
- 不影响主会话的对话
- 主会话和定时任务完全分离

**应用到我们的场景**：
- 内容创作团队的工作流 → 交给主编在独立会话中执行
- 首席助理只负责与陛下对话
- 两者互不干扰

---

### 2. Delivery投递机制
**借鉴点**：
- 明确的delivery配置
- 推送到指定channel和to
- announce模式主动通知

**应用到我们的场景**：
- 主编完成任务后 → 通过delivery主动推送给陛下
- 不需要陛下询问
- 确保陛下能收到结果

---

### 3. 心跳检查兜底机制
**借鉴点**：
- 每2小时检查一次
- 检查lastRunStatus
- 检查deliveryStatus
- 检查consecutiveErrors

**应用到我们的场景**：
- 我们的心跳检查改为：每小时检查定时任务状态
- 检查任务是否按时启动
- 检查汇报是否正常
- 发现问题立即修复

---

### 4. 三层闹钟 + 独立会话
**结合我们已有的方案**：

| 机制 | 作用 |
|------|------|
| **三层闹钟** | 提前10分钟、准时、5分钟后检查 |
| **独立会话** | 主编在独立会话中执行，不影响主对话 |
| **Delivery投递** | 任务完成后主动推送给陛下 |
| **心跳检查** | 每小时检查任务状态，兜底保障 |

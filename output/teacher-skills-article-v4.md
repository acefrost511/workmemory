# 教师的日常工作，哪些值得做成 AI skill？

上周，AI 圈和很多 AI 交流群都被一个 GitHub 项目刷屏了——"同事.skill"。简单说：把离职同事、导师、搭档的聊天记录和工作文件喂给 AI，立刻炼出一个能 24 小时干活的数字分身。

比如离职的程序员，他的代码规范、甩锅姿势、技术判断风格，都能保持原汁原味。有人感叹"赛博永生"，有人苦笑"我的牛马经验也要给公司永远打工了"。

"skill"这个词，你可能听过但不一定清楚它为什么让 AI 圈这么兴奋。

打个教学场景的比方：大模型就像一个学历很高的实习老师——学科知识全都学过，理论张口就来，但一站上讲台面对具体的班级和学情，它缺的是判断力。skill 就是给这个实习老师装上一套老教师的判断系统——里面封装了做某件事的完整经验、流程和质量标准。装上之后，它就不再是泛泛而谈的通才，而是在特定任务上有章法、有标准的专才。

## 一、skill 和提示词 prompt 有什么区别？

很多老师已经在用 AI 了，还能非常熟练地通过提示词 prompt，用豆包或 Kimi 生成教案初稿，或者批量命题。而 skill 的底层也是基于自然语言驱动的，所以很多人第一反应就是"skill 不就是个写得好的 prompt 吗？"

表面上看确实像，但它们之间至少有三层本质差异。

**第一层差异：形态不同——prompt 是一段话，skill 是一个"工具包"。**

Prompt 就是你在对话框里输入的一段指令。而 skill 在 Claude 的技术框架里，是一个完整的目录结构。它的核心是一份叫 SKILL.md 的文件，开头有一张"名片"（技术上叫 YAML 元数据），写着这个 skill 叫什么、什么场景下应该被调用；后面是用 markdown 写的操作指南。除了这份核心文件，skill 还可以附带脚本（帮 AI 执行重复性操作）、参考资料（供 AI 按需查阅）、模板文件（直接用于输出）。你可以把 prompt 理解为"一张便签纸上的口头交代"，skill 理解为"一个装了说明书、工具和参考手册的工具箱"。

**第二层差异：调用方式不同——prompt 是你手动粘贴的，skill 是 AI 自己判断要不要用的。**

Prompt 需要你每次手动输入或粘贴。而 skill 有一个主动触发机制：AI 看到你的请求后，会自动扫描所有可用 skill 的"名片"描述，判断哪个 skill 适合处理你当前的任务，然后自动加载并执行。这意味着你不需要记住"我上次那段提示词怎么写的"，只要 skill 的触发描述写得准确，AI 遇到相关任务就会自动调用它。

**第三层差异：生命周期不同——prompt 是一次性的，skill 是可迭代的工程。**

Prompt 写完就用了，下次效果不好你只能重新写。skill 有一套完整的开发和验证流程：先写草稿，然后造几个测试案例，让 AI 分别在"有 skill"和"没有 skill"的情况下各跑一遍，对比产出质量；再结合人工评审和量化打分反复调优，直到效果稳定。改完一版，测试数据和历史版本都保留，像软件一样做版本管理。

在这三层差异之上，skill 还有几个 prompt 不具备的实质优势。**可复现**：同一个 skill 不同人用，产出质量是稳定的，因为判断标准和执行流程封装在里面了；prompt 则高度依赖你当时怎么措辞。**可组合**：多个 skill 可以串联，比如"学情分析 skill"的输出直接喂给"教学设计 skill"作为输入，形成工作流；prompt 很难做到这种模块化衔接。**渐进加载**：skill 有三层加载机制——AI 平时只读"名片"（约 100 词），触发后才加载操作指南（建议不超过 500 行），附带的参考资料则按需调用。这意味着 skill 不会白白占用 AI 的注意力资源。

打个比方：prompt 相当于你口头跟代课老师交代"这节课大概这么上"，skill 相当于你给他一个备课工具箱——里面有备课手册、常见学情速查表、优秀教案模板、还有一份"什么情况下翻哪一页"的索引。前者靠临场发挥，后者靠系统执行。

回到前面那个同事.skill 和赛博永生的故事，虽然让人想笑但又感觉有点可悲，这让我想到另一个问题：程序员同事离职了，他的代码还能跑。那老师呢？教了三十年书的骨干教师退休了，他们的教法谁来传承？

这个问题值得认真想一想。但在那之前，还有一个更基础的问题：你知道老师日常做的哪些工作，其实最值得做成 skill 吗？

## 二、老师的工作，哪些值得做成 skill？

不是所有教学环节都适合 skill 化。一个简单的区分方法：看这件事有没有相对固定的输入和输出？有没有可复用的最佳实践？如果有，做成 skill 的价值就高。

下面是一张按教学流程整理的任务全景图。

### 2.1 日常教学设计

**新授课设计**——输入学科、年级、单元、课时目标，输出完整教案（导入→新知→练习→总结）。skill 化价值最高（⭐⭐⭐）。

**复习课设计**——输入单元知识点清单和学情数据，输出复习路径与重点题目。skill 化价值最高（⭐⭐⭐）。

**试卷讲评设计**——输入试卷和班级成绩分布，输出讲评顺序、重点题和补偿练习。skill 化价值高（⭐⭐）。

**作业设计**——输入课时目标和预估完成时间，输出分层作业清单。skill 化价值高（⭐⭐）。

### 2.2 学情判断

**学情诊断**——输入最近一次考试或作业数据，输出错误分类统计和认知薄弱点定位。skill 化价值最高（⭐⭐⭐）。

**错题归因**——输入学生错误样本，输出错误类型判断（计算性、概念性还是策略性）。skill 化价值最高（⭐⭐⭐）。

**前置知识检测**——输入下一个单元的学习目标，输出诊断题目清单和通过标准。skill 化价值高（⭐⭐）。

### 2.3 教研协作

**集体备课辅助**——输入备课主题和学情背景，输出讨论框架、预设问题和参考教案。skill 化价值高（⭐⭐）。

**教案互审**——输入待审教案，输出多视角审阅意见，分为必须修改、建议修改和可选优化三级。skill 化价值最高（⭐⭐⭐）。

**试卷命制分析**——输入双向细目表和预期难度，输出难度评估、区分度预测和修改建议。skill 化价值中等（⭐）。

## 三、四个核心 skill 详解

上面每项任务都可以单独做成 skill。但制作 skill 需要投入时间精力，优先级最高的是那些高频、标准化程度高、对教学质量影响大的任务。

以下四个同时满足这三个条件，是教研组最值得优先动手的。

在展开之前，先说明一下：一个 skill 的核心结构是一份 SKILL.md 文件。开头是元数据区（技术上叫 YAML frontmatter），写明 skill 的名称和触发描述——触发描述决定了 AI 在什么情况下会自动调用这个 skill。后面是操作指南主体，用自然语言写成。

下面每个 skill，我会先展示输入输出的实际效果，再给出 SKILL.md 的核心内容。需要强调的是：这些 SKILL.md 里的指令不是写给人看的操作手册，而是写给 AI 看的——所以它更注重解释"为什么这样做"，而不是堆砌"必须""禁止"等刚性规则。Claude 的 skill 设计哲学认为，AI 有足够的理解力，给它讲清楚背后的道理，比给它下死命令更有效。

### Skill 1：lesson-designer（新授课设计）

**一句话说清楚**：输入学科、年级、单元和课时信息，输出符合教研组标准的一节完整教案。

**输入示例**：

> 学科：高中数学｜年级：高一｜单元：三角函数｜课时：诱导公式（第一课时）
> 课时目标：学生能运用诱导公式求任意角的三角函数值
> 班级学情：中等水平，运算能力薄弱，前置知识"单位圆与象限角"掌握率约 70%

**输出片段**：

【导入环节】（建议 5–7 分钟）

目标：激活单位圆知识，为诱导公式引入做铺垫。

① 黑板画单位圆，快速点名回答四个象限的坐标特征（2 分钟）。

② 追问："如果把角写成 (π/2+α)，点在单位圆上的位置会怎么变？"引出本节核心问题（2 分钟）。

③ 板书课题，说明本节要解决的核心问题：如何把任意角的三角函数值转化为锐角的三角函数值（1 分钟）。

【后续：新知建构→例题→变式→总结，完整输出】

**SKILL.md 核心内容**：

```yaml
---
name: lesson-designer
description: |
  设计一节新授课的完整教案。当用户提到备课、写教案、设计新课、
  准备课时方案，或者给出了学科/年级/课题信息并希望生成教学设计时，
  使用这个 skill。即使用户没有明确说"教案"二字，只要意图是
  设计一节具体的课，都应该触发。
---

markdown
# 新授课设计 skill

你的任务是帮助教师设计一节完整的新授课教案。

## 为什么这个 skill 存在

一线老师备课时间有限，但一节好课的设计需要同时兼顾教学目标、学情适配、
环节衔接和时间分配。这个 skill 把有经验的教研组的设计逻辑封装起来，
帮助教师快速生成一份"能用的初稿"，再根据自己的判断调整。

## 你需要从用户那里获取的信息

开始设计前，确认以下信息。如果用户没有提供，主动询问——
缺少关键信息会导致教案泛泛而谈，这比不写还糟糕。

- 学科、年级、教材版本
- 单元/章节、课时主题
- 课时目标（最好用"学生能……"的句式，如果用户给的目标不够具体，
  帮他转化为可观测的行为描述，并确认）
- 班级学情（如果用户不确定，可以引导他回忆：上一次相关测试的大致表现、
  学生在前置知识上的薄弱点）

## 设计逻辑

一节 40 分钟的新授课通常包含五个环节。这不是刚性模板——
如果课型特殊（如实验课、探究课），可以灵活调整，
但需要说明调整的理由。

**导入（5–7 分钟）**：导入的核心目的是激活前置知识、建立认知连接。
常见的失败模式是"为了有趣而有趣"——花 5 分钟讲了一个故事但和新知
没有认知关联。好的导入应该让学生带着一个具体问题进入新知环节。

**新知建构（10–15 分钟）**：这是最考验设计功力的环节。
关键原则是"让学生经历知识的发生过程"，而不是直接给结论。
设计时要想清楚：概念从哪里引入？关键的认知转折点在哪？
学生可能在哪一步卡住？卡住时用什么追问或支架帮他跨过去？

**例题讲解（5–8 分钟）**：例题的选择需要说明理由——为什么选这道而不是
课本上的另一道。讲解步骤中要设计"等待时间"：
给出题目后不要立刻讲解，让学生先想 1–2 分钟。

**变式练习（8–10 分钟）**：至少设计 3 道变式，变式之间要有明确的
梯度递进逻辑（改了什么条件？为什么这个变化能检验更深层的理解？）。
避免 3 道题本质上考的是同一个东西。

**总结与作业（3–5 分钟）**：总结不是教师复述知识点，
而是让学生用自己的话说出"今天学了什么、和之前学的有什么关系"。
作业建议分层：基础巩固层、能力提升层、挑战拓展层。

## 学情适配

这是这个 skill 最重要的部分。没有学情适配的教案是通用模板，不是教学设计。

- 如果用户提供了学情信息，所有环节的设计都要显式回应学情。
  比如"运算能力薄弱"→ 例题讲解中增加运算步骤的拆解，
  变式练习中设计专门针对运算错误的纠错题。
- 如果学情信息不完整，基于一般学情做设计，但必须在输出末尾
  列出你做了哪些假设（不超过 3 个关键假设），请用户确认。
  假设被否定后，相应环节要调整。

## 输出格式

按上述五个环节依次输出。每个环节标注建议时长和设计意图。
课时总长默认 40 分钟。
```

你可以看到，这和一段 prompt 有明显的区别：它有触发描述（AI 知道什么时候该调用它）、有设计理念的解释（AI 理解"为什么"而不只是"怎么做"）、有对失败模式的预警（AI 知道什么是常见的坑）。这不是一段一次性的指令，而是一份可以反复调用、持续迭代的工作指南。

### Skill 2：lesson-reviewer（教案预审）

**一句话说清楚**：把一节课的教案丢进去，模拟教研员、学优生、学困生三个视角同时审阅，输出"必须修改 / 建议修改 / 可选优化"三级意见。

这是整个系列里价值最直接的一个——年轻老师写完教案，提交前过一遍这个 skill，相当于请三个人同时帮你把了一道关。

**输入示例**：

以下是一份高一数学"三角函数诱导公式"第一课时的教案，请从多个视角审阅。

【导入】（5 分钟） 提问：sin30° 等于多少？cos60° 呢？有什么关系？教师口述：sin30°=1/2，cos60°=1/2，所以 sin30°=cos60°。引出课题：今天我们学习诱导公式，看看角的度数之间的关系。

【新知建构】（12 分钟） 直接给出公式：sin(90°-α)=cosα，cos(90°-α)=sinα。板书推导过程，教师讲解。举例验证：令 α=30°，sin(90°-30°)=sin60°=√3/2，cos30°=√3/2。

【例题】（10 分钟） 例 1：求 sin(90°-25°) 的值。讲解步骤：先代入公式→再计算。例 2：求 cos(90°-60°) 的值。学生活动：同桌互相出一道类似的题并交换解答。

【练习】（10 分钟） 给出 5 道练习题，全是"90°-α"型。学生独立完成，教师巡视。答案全班统一订正。

【总结】（3 分钟） 教师总结：本节我们学习了两个诱导公式，需要熟练运用。作业：练习册第 28–30 页。

**输出片段**：

**教研员视角：**

导入环节用了 5 分钟，但全程是教师讲授，没有明确的学情诊断依据来支撑"学生已掌握弧度制和象限角"这一判断。建议在导入环节增加一个 30 秒的前置检测——如"请在单位圆上标出 α=30°、60°、90° 的位置"，根据举手人数判断是否需要补充弧度制知识。【必须修改】

新知建构直接给出公式，缺少学生自主发现的过程。诱导公式的本质是"角度加减 π/2 后函数值的变化"，学生如果不知道这个几何直观，直接背公式会很快遗忘。建议增加"单位圆上点的旋转动画"或"用坐标纸描点"的探究活动。【必须修改】

练习题全部是同一题型（90°-α），无法检测学生是否能识别不同的变式方向。【建议修改】

**学困生视角：**

例题从"90°-α"直接跳到了一般角的诱导公式，中间没有过渡题。我在"90°-α"这里还没完全搞清楚（尤其是"角度相减对应点怎么转"），直接看到一般角的公式会懵。建议在给出一般公式之前，先用几个具体的 90°-α 的题目让我找到规律，而不是直接告诉我结论。【必须修改】

变式练习 5 道题全是"已知角求函数值"，没有反向题（已知函数值求角的范围，或者已知函数值判断角的可能取值）。这类题我经常出错，但练习里完全没有涉及。【建议修改】

**学优生视角：**

整节课的挑战性不够。诱导公式还有 sin(180°-α)=sinα、cos(180°+α)=-cosα 等多组，全部只讲了一组就进入练习，深度没有拉开。建议在学生完成基本练习后，增加一道综合题：sin(90°+30°) 怎么求？引导学生发现还有更多的诱导公式需要探究。【建议修改】

例题讲解之后有"同桌互出题"的活动，这个设计很好，但时间只给 2 分钟就进入统一订正，实际上学生还没有充分讨论。建议延长至 4–5 分钟，并增加"谁的题最有挑战性"的展示环节。【可选优化】

**SKILL.md 核心内容**：

```yaml
---
name: lesson-reviewer
description: |
  从多个视角审阅一份教案，输出分级改进意见。当用户提交了一份教案并希望
  获得反馈、审阅、改进建议，或者说"帮我看看这个教案""这节课设计有没有
  问题""教案预审"等，都应该触发。也适用于用户刚用 lesson-designer 生成
  了教案，紧接着想检查质量的场景。
---

markdown
# 教案预审 skill

你的任务是从多个视角审阅一份教案，帮助教师在正式上课前发现问题。

## 为什么多视角审阅很重要

一份教案的问题往往藏在"教师觉得理所当然、但学生实际体验完全不同"的
缝隙里。教研员看结构和规范，学困生暴露理解断点，学优生暴露深度不足。
三个视角同时审阅，覆盖面远超单一角色的反馈。

## 三个审阅视角

**教研员视角**——关注教学设计的专业性：
教学目标是否清晰且可观测？有没有前置知识诊断的依据？
教学环节是否完整且衔接合理？时间分配有没有依据？
是否存在"教师讲得很顺但学生其实没有参与"的环节？
这个视角最容易发现的问题是：教案写了很多"教师做什么"，
但很少写"学生在做什么、怎么判断学生是否跟上了"。

**学困生视角**——模拟班级中后段学生的体验：
哪个环节会让我卡住？例题到变式的跳跃是否太大？
概念引入有没有足够的直观支撑（图示、类比、操作）？
这个视角的价值在于：教师自己做过一遍觉得"很简单"的地方，
恰恰可能是学困生的断崖。

**学优生视角**——模拟班级前段学生的需求：
这节课有没有让我觉得"太简单了没意思"的地方？
有没有深挖空间和探究延伸？
好的教案应该让学优生"吃得饱"但不偏离主线。

## 意见分级标准

**必须修改**：会直接影响教学效果的问题——比如新知环节缺少学生参与、
例题和变式之间存在认知断崖、时间分配明显不合理。
这些如果不改，课大概率会出问题。

**建议修改**：改了会更好，不改也能勉强达到目标——比如某个环节
可以设计得更精巧，某道练习可以换一道更有区分度的。

**可选优化**：锦上添花的改进空间——比如增加一个展示环节、
调整某个活动的时间让节奏更从容。

分级时要诚实。不要为了显得"有用"而把小问题升级为"必须修改"，
也不要为了显得"宽容"而把真正的问题降级为"可选优化"。

## 输出格式

按视角分段输出，每个视角独立列出意见，每条意见标注优先级。
意见要具体到"哪个环节、什么问题、为什么是问题、建议怎么改"，
不要说"建议优化导入环节"这种空话。
```

### Skill 3：test-analyser（试卷与学情分析）

**一句话说清楚**：输入一份试卷和班级成绩数据，输出错误分类统计、各知识点掌握率，以及下一节课可用的补偿练习建议。

**输入示例**：

> 试卷：高一数学单元测试（三角函数）｜班级均分：72.3｜年级均分：68.5
> 得分率低于 60% 的题目：第 5 题（10 分）、第 11 题（12 分）、第 13 题（10 分）
> 第 5 题典型错误：多数学生写了 sin(90°-α)=cosα，但符号判断错误
> 第 11 题典型错误：学生不知道什么时候用 sin(π/2+α)，什么时候用 sin(π-α)

**输出片段**：

【整体评估】班级均分高于年级均分 3.8 分，表现良好。但标准差偏大（14.2），说明班级内部分化严重。

【核心问题定位】

- 符号判断错误（第 5 题，错误率 62%）：认知根源是对"90°+α"和"90°-α"在单位圆上的位置变化方向混淆。
- 公式选择错误（第 11 题，错误率 54%）：认知根源是不理解"奇变偶不变"口诀背后的几何含义。
- 综合应用缺失（第 13 题，错误率 71%）：无法将多个诱导公式组合使用。

【补偿练习建议】

针对第 5 题（符号判断）：设计"纠错型"练习，给出 3 道符号错误的解题过程，让学生指出错误并改正。例："sin(90°+30°)=sin60°=√3/2，请指出错误并给出正确答案。"

针对第 11 题（公式选择）：设计"分类选择"练习，给出 10 道题，每题包含两个选项（分别对应不同公式），学生选择正确公式并说明理由。

【下一节课教学调整建议】建议在下一节课开始前增加 5 分钟"诱导公式家族图谱"整理，用结构化板书帮学生建立完整认知框架，而非零散的公式记忆。

**SKILL.md 核心内容**：

```yaml
---
name: test-analyser
description: |
  分析试卷成绩数据，定位学生的知识薄弱点，并给出补偿教学建议。
  当用户提到考试分析、成绩分析、错题分析、试卷讲评准备、学情数据解读，
  或者提供了班级成绩数据希望获得教学建议时，使用这个 skill。
  也适用于用户拿到一份试卷想了解"学生到底哪里没学好"的场景。
---

markdown
# 试卷与学情分析 skill

你的任务是基于试卷和成绩数据，帮教师定位学生的认知薄弱点，
并给出可操作的教学调整建议。

## 为什么这个 skill 需要"透过数据看认知"

教师拿到成绩数据，最直观的反应是"哪道题错得多"。
但"错得多"只是现象，教师真正需要的是"为什么错"和"下一步怎么教"。
这个 skill 的核心价值是把"错误数据"翻译成"认知诊断"再翻译成"教学行动"。

## 分析逻辑

**第一步：整体画像。** 班级均分在年级中的位置、班级内部的离散程度（标准差）。
离散程度很重要——均分相同但标准差不同的班级，教学策略完全不同。
标准差大意味着分化严重，需要分层；标准差小意味着整体水平接近，
可以统一推进。

**第二步：定位高错率题目的认知根源。** 不要停留在"这道题错误率高"，
要追问：学生犯的是什么类型的错误？是计算性错误（会做但算错）、
概念性错误（根本没理解）、还是策略性错误（理解了但不知道什么时候用）？
不同类型的错误，补救方式完全不同。

如果用户提供了典型错误样本，直接基于样本分析。
如果没有，根据题目特征和错误率推断最可能的错误类型，
并标注"这是基于推断的判断，建议用实际错题样本验证"。

**第三步：生成补偿练习建议。** 每类高错率问题对应一种练习形式。
原则是"不要重复练同类型的题"，而是用不同形式攻击同一个认知薄弱点。
比如符号判断错误，不是再做 10 道同类题，而是做"纠错型"练习
（给出错误解题过程让学生找错），这对修正认知更有效。

**第四步：教学调整建议。** 基于本次数据，下一节课应该做什么调整。
这是教师最需要的"可操作结论"。

## 诚实边界

分析结束时，主动说明哪些结论是数据直接支持的，
哪些是基于经验推断的。数据有限时不要过度解读——
一次考试的数据能告诉你的东西是有限的，
承认这一点比给出虚假的确定性更有价值。
```

### Skill 4：study-diagnoser（学情前置检测）

**一句话说清楚**：输入下一个单元的学习目标，输出一份可嵌入课堂前 5 分钟的诊断题清单和判断标准，告诉你学生进教室前已经知道了什么、还不知道什么。

这个 skill 是前面三个的基础设施——lesson-designer 需要学情信息才能做好适配，lesson-reviewer 需要知道学生起点才能准确审阅，test-analyser 的分析也需要前测数据做对照。学情前置检测为它们提供数据入口。

**输入示例**：

> 学科：高中数学｜年级：高一｜下一单元：平面向量（加减法与数乘运算）
> 学生即将学习的内容：向量的定义与表示、向量加法的三角形法则与平行四边形法则、向量数乘运算的几何意义
> 班级学情背景：刚学完三角函数，运算能力中等，空间想象能力个体差异大

**输出片段**：

【前置知识检测清单】

**题 1**：画一个直角坐标系，标出点 A(3,2) 和点 B(5,4)，向量 AB 怎么表示？
——对应先备知识：有向线段的概念。通过标准：80% 学生能正确表示。不通过时的补救建议：补充有向线段定义，用物理位移做类比。

**题 2**：计算 3×(2,1) 和 (2,1)×3，结果相同吗？
——对应先备知识：标量乘法基本运算。通过标准：75% 学生能正确计算。不通过时：用温度变化类比标量乘法。

**题 3**：在坐标系中画一条从 (0,0) 到 (3,4) 的箭头，再画一条从 (3,4) 到 (6,8) 的箭头，两条箭头方向相同吗？长度有什么关系？
——对应先备知识：方向与比例直觉。通过标准：60% 学生能直觉判断。不通过时：这是本单元的核心障碍点，建议在引入环节重点处理。

【关键发现】本班学生在"向量方向"上的直觉较强（题 3 通过率高于预期），但在"向量坐标表示"上掌握率偏低（题 1 仅约 70%）。建议：引入环节可以从方向入手（题 3），再过渡到坐标表示（题 1），最后抽象为数乘运算。

【检测时间估算】题 1–3 共需约 5–7 分钟。建议嵌入课堂前 5 分钟，不单独占用一节课。

**SKILL.md 核心内容**：

```yaml

description: |
  为即将开始的新单元设计学情前置检测题。当用户想了解学生的知识起点、
  想在新课前做一个快速摸底、或者说"学生的基础够不够学这个单元""需要
  补什么前置知识"等，使用这个 skill。也适用于用户提到"课前诊断""学前
  测试""前测"等场景。

# 学情前置检测 skill

你的任务是帮教师设计一份嵌入课堂前 5 分钟的快速诊断工具，
判断学生是否具备进入新单元所需的前置知识。

## 为什么前置检测比"凭经验估计"更可靠

很多教师习惯凭教学经验判断"学生应该会了"。但实际上，
不同班级、不同年份的学生基础差异很大，上一届会的东西这一届不一定会。
5 分钟的前置检测成本极低，但能避免"讲了半节课才发现学生根本没跟上"
这种代价极高的情况。

## 设计逻辑

**确定先备知识清单**：根据用户提供的新单元学习目标，
逆向推导学生需要哪些已有知识作为支撑。通常 3–5 个就够了。
每个先备知识对应一道检测题。

**检测题的设计原则**：
每道题只测一个先备知识——混合测多个知识点的题无法精确定位问题。
答案有明确的对错标准——不要用开放性问题，因为 5 分钟内要快速判断。
难度略低于"刚好够用"——前置检测不是考试，目的是发现"连基本的都不会"
的情况，不是区分优良中差。

**设置通过标准和补救方案**：每道题标注预期通过率。
如果实际通过率低于预期，给出两种应对：
一是"可以在导入环节花 3 分钟快速补救"的轻量方案；
二是"这个缺口太大，需要单独用一节课补前置知识"的重度方案。
教师根据实际数据选择。

## 时间约束

整套检测题必须能在 5–7 分钟内完成。
这是硬约束——超过这个时间就会挤压正课时间，教师不会用。
题目数量通常控制在 3–5 道。

## 输出中要包含的内容

检测目标（测什么先备知识）、检测题清单（每道题标注对应的先备知识、
通过标准、未通过时的补救建议）、结果解读指南（不同通过率对应什么
教学决策）。

如果检测结果显示学生在某个先备知识上存在普遍缺失，
主动建议教师调整新课的引入顺序——从学生已有的强项入手，
再过渡到薄弱点，而不是按教材顺序硬讲。
```

## 四、怎么判断一个教学经验值不值得做成 skill？

上面四个 skill 可以直接照做。但如果你想让教研组自己生产更多 skill，还需要一把尺子——什么值得做，什么不值得？

一个教学经验要值得封装成 skill，以下三个问题都得答"是"。

**问题一：换个单元还管用吗？**

检验的是这条经验有没有跨情境的通用性。比如"例题讲完先让学生出题再做变式"这个策略，在三角函数、几何证明、概率统计三个不同单元都被验证过有效——它是一个可复用的教学策略，值得封装。但如果某条经验只在某道特定题上管用，它就只是一条操作建议，写进提示词就行，不必单独做 skill。

这也呼应了 skill 的设计哲学：skill 的目标是被反复调用，适用于不同的输入和场景。如果一条经验只对特定案例有效，做成 skill 反而会"过拟合"——在那个案例上表现完美，换一个就失灵。

**问题二：能帮你预判新课怎么教吗？**

检验的是这条经验有没有预测力。比如"学生在 A 知识点犯的错误类型，往往会在 B 知识点重演"这个判断模式，不仅能解释过去的错误，还能预测学生在新单元哪里会卡住——它对未来的教学设计有指导价值。如果一条经验只能解释已经发生的事，对下一步该怎么做没有启发，它的 skill 化优先级就低。

**问题三：是不是只有一部分老师才这么做？**

检验的是这条经验有没有"增量价值"。"课前要写教案"不需要做成 skill——所有老师都这么做，封装了也没有信息增量。但"例题和变式之间插一个学生出题环节"就不同——大多数老师讲完例题直接上变式，这个差异才是值得显性记录的教学判断。

三个都满足的，值得投入时间做成 skill。只满足一两个的，降级为操作建议直接写进提示词。一个都不满足的，不收录。

## 五、教研组怎么协作制作和维护 skill？

skill 不是一个人埋头做出来的。一个有生命力的 skill，需要教研组集体参与，而且需要经历一个"写→测→评→改"的迭代过程。

分工可以这样安排：主笔负责提炼自己的教学经验，提供最佳样本——比如你设计过的最好的教案、最有效的学情分析记录；参与者在集体讨论中提供不同视角，比如不同班级、不同学情下的反馈；审核者负责检验 skill 产出的判断逻辑是否与教研组共识一致。建议在每个 skill 的文档里注明主笔和参与讨论的成员，尊重贡献者的劳动。

开发流程建议分四步走。第一步，先写一份 SKILL.md 草稿，把教学经验和判断逻辑用自然语言写进去。写的时候记住一个原则：多解释"为什么这样做重要"，少堆砌"必须这样、禁止那样"的刚性规则——AI 不是流水线工人，它有理解能力，讲清道理比下死命令更有效。第二步，造 2–3 个测试案例，选教研组已经做过设计的真实场景。第三步，让 AI 在"有 skill"和"没有 skill"的情况下分别跑一遍同样的任务，对比两组输出的质量差异——这一步至关重要，它能告诉你 skill 到底有没有带来实质提升，还是只是"看起来更规范但没有真正变好"。第四步，根据对比结果调整 skill 内容，重复这个循环，直到输出稳定达标。

更关键的是，skill 是活的，不是写完就归档。建议每学期至少回顾更新一次，触发条件包括：教材或课标发生变化、接了一个学情明显不同的新班级、使用中发现判断逻辑有偏差。更新时保留历史版本并标注修改原因，方便追溯。

一个 skill 达到"可用"的最低标准需要三样东西：有明确的触发条件（AI 和教研组其他成员都知道什么时候该用它），有可验证的输出（和实际教学效果对比方向一致），有清晰的边界（什么场景下它会失效，已经标注出来）。

## 六、skill 做完之后，还需要验证

写完一个 skill 不是终点，你需要测试它。这里提供四种由简到繁的测试方法。

**对照测试**：这是最核心的一步。选一个教研组已经做过设计的单元，让 AI 分别在"加载了 skill"和"不加载任何 skill"的情况下，针对同一个任务各跑一遍。对比两组输出：加载 skill 之后，教案的学情适配度、环节衔接逻辑、判断合理性是否有明显提升？如果两组输出差异不大，说明 skill 里封装的经验还没有真正发挥作用，需要回去调整。

**已知答案测试**：选教研组公认设计得好的一节课，让加载了 skill 的 AI 重新做一遍，和老教师的实际设计对比。不要求完全一致，但大方向应该对齐——如果老教师觉得"这个导入环节抓住了关键"，AI 的输出也应该在导入环节体现类似的判断。偏离严重就回去调整判断规则。

**边缘测试**：选一个教研组没有人做过的新题型或新课型，让 skill 推断应该怎么做。合格的表现是：AI 能给出合理的推断，同时会标注"这部分基于推断，不确定是否适用"。如果它斩钉截铁给了一个答案，说明 skill 的"诚实边界"没设好——它应该知道自己不知道什么。

**风格测试**：让 skill 输出一段 100 字的教学建议，读一读：有没有你们教研组的味道？还是通用的 AI 鸡汤？还是教科书原文的拼凑？好的 skill 应该带着你们团队自己的教学理念和话语风格。

四种测试不需要一次性全做。建议至少做前两种（对照测试和已知答案测试），这两种能覆盖最基本的质量检验。后两种在 skill 稳定使用一段时间后再做，用于发现潜在的盲区。

## 结语

回到开头的问题：老师的日常工作，哪些值得做成 skill？

不是所有工作都值得。最值得做的是那些高频发生、有标准流程、有最佳实践可循、但目前全靠个人经验支撑的环节——新授课设计、教案预审、学情分析、学前诊断。

做好这四个，你的教研组就有了一套可运行的"教学智囊团"。年轻老师备课的时候随时可以问一句：教研员会怎么评价这个导入？学困生在这个环节会卡在哪里？这道例题的变式梯度够不够？

每个教研组都有那么一位老师，大家习惯了有问题就去问她。但很少有人想过，她的那些判断——什么时候该补基础、什么时候该拉难度、什么时候学生的沉默是没听懂而不是在思考——这些东西有一天会随着她的离开而消失。

skill 不能替代她。但它能让她最有价值的那部分经验，留在教研组的工具里，被后来的人继续使用、继续修订、继续生长。
